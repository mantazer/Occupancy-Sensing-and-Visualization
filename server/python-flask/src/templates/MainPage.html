Project Pulsar, Reporting Power Outages

Fall 2014, [PennApps X]

During large power outages, especially those caused by large storms, power suppliers waste lots of money and time trying to figure out what houses and neighborhoods have power and which don't. When Hurricane Sandy hit the New York/New Jersey area, it was estimated that power suppliers took thousands of trips out to the affected areas to simply figure out which houses had power and which ones didn't. This massive inefficiency was the inspiration for Project Pulsar. My partner Muntaser and I created a system that rests on the basic assumption that as long as a house has access to internet, that they have access to electricity. We used an Intel Galileo Embedded System to send http requests to a central Amazon EC2 server, with the idea that the embedded system represents the "sensor" in the house. With the server, we had a central database that kept track of the state of every embedded system on the network. When the server notices that one of the systems has been down longer than usual (meaning an http request hasn't been received from the embedded system recently), we probe the Internet Service Provider (i.e. Comcast) via a Twilio Application in order to figure out if there is an Internet Outage on the side of the ISP as opposed to a power outage. Once this decision is made, and if it is determined that there is a Power Outage, the information is conveyed to the power distributor via a web client.

This project gained a lot of recognition at PennApps, especially by the hardware sponsors. Both Muntaser and I, as well as a couple of Electrical Engineering friends, hope to continue the project to see if we can truly find a way to determine power outages cheaply and efficiently. Although Project Pulsar as it stands may fit the basic mold, getting access to user's internet (with security and passwords) as well as configuring with the Internet Service Provider (so as to differentiate between a power outage and internet outage) has to be fleshed out more before this project can truly become scalable.

If you are interested in finding out more about the project, feel free to shoot me an e-mail .

OneDir, Rudimentary Version of DropBox

Spring 2014 [Advanced Software Development Methods]

The problem statement here was pretty simple. Using Python, we were tasked with creating our own rudimentary version of DropBox. The problem can be broken down into two separate sections, the client side which has to recognize file changes and "ping" the server accordingly and the server side which has to take into account these changes and update the other synchronized users accordingly. For the client side, we used the Python framework WatchDog in order to monitor files. WatchDog is extremely easy to use, providing you the ability to trigger events based on the file activity that happened (for example, you can have a different event triggered if a file is created versus deleted). On the sever side, we built a Flask web server (again, very easy to use) and utilized its file transfer libraries in order to retrieve the updated files. At a basic level, we had each triggered WatchDog event hit a different endpoint of the Flask webserver so that the server could act accordingly. We also had another thread working in the background monitoring the server in case changes happen on another client's end (so that all machines that are accessing the server are updated). The Flask server was run on an AWS micro-instance.

Although I worked in a group of three other students on this project, I can confidently say that I did 95% of the coding for this project (as highlighted by this graph). Even though it wasn't the most ideal situation, especially during crunch time at the end of the semester, I really learned a lot from the project as a whole. From my experience building this OneDir system from back to front, I got comfortable picking up new libraries and frameworks that I knew nothing about and figuring out how to use them. I got really good with my debugging, google searching, and just general understanding of what to look for when figuring out new things.



TrashTalk, Android Application

Geography Toy for Elementary Aged Students

Fall 2011 [Introduction to Engineering Class]

My first real, overwhelming introduction to engineering and college came in the form of my first engineering class, ENGR 1620 (Introduction to Engineering). For the final project of the semester, me and a group of four other students were tasked with making a physical, educational toy for elementary aged students. As easy as this sounded, the overall scope of the project became apparent very quickly. Coordinating the programming of the toy with the physical aspect, while also making sure to fulfill the proper standards of learning in Virginia and including feedback from multiple sources, was extremely challenging. Although it there were some of bumps in the road and a lot of sleepless nights, I learned a lot from the project in the end.

On the programming end, the entire project was built in JAVA. Our toy was a Geography Toy that asked questions related to continents for students to answer. The students could physically select a continent by moving an arrow over what they believed the answer was (we coordinated with RFID tags in order to realize which continent was which). A GUI with a picture of the World Map, the question, and a students points was shown on a computer screen. As the student answered questions, the questions generated would be adjusted based on the percieved level of the student.

Unfortunately, I did not know the magic of GitHub or the importance of neat code at this point in my coding career. The code is really hard to follow and is very honed into the problem that we had. If you have any questions about the code, feel free to e-mail me.

Exploring the Use of Student Taught Classes to Introduce New Technical Topics to Engineering Undergraduates

With an enrollment of more than 50 students on campus, and over 1,000 unique visitors online, we knew that we had something special. About The Site



So what's under the hood...




"Success is a journey, not a destination. The doing is often more important than the outcome."

- Arthur Ashe


If you haven't seen it already, the code for this site can be found here. This site is built using a Python Framework called Flask. For anyone wanting to build their own site from scratch, I highly recommend using Flask. It is extremely easy to use and very intuitive. From a server standpoint, this site is running on Google App Engine. Although the App Engine documentation makes it sound like it is very easy to run a Flask webserver on their machines, I did not find this to be the case. I spent many hours trying to figure out how to run Flask on Google's platform, and even considered jumping to Amazon Web Services a couple of times. One of the biggest problems is that unless the 3rd party application you want to use is on this list, you have to physically download these files and put them in the right hierarchy among your other folders and files related to the project. This got especially annoying with Flask, because Flask is built on other un-supported 3rd party libraries such as werkzeug and jinja2. Eventually I was able to get the web server working, then it was a matter of figuring out the HTML and CSS. For those of you who aren't interested in learning how to use Flask with Google App Engine, or don't want to deal with a webserver at all, I recommend looking into GitHub Pages. GitHub Pages is perfect for those people who are more worried about the HTML, CSS, and JavaScript because it abstracts away the Web Server.

Based upon the troubles I had, and the amount of searching I had to do to find the right answers, I figured it would be worthwhile to create a step-by-step tutorial on how to build a Flask webserver on App Engine. Although there are ready-to-go GitHub repositories that you can simply fork and use, I did not find this as satisfying as figuring it out myself. There were a couple of solutions I happened on that turned out to be outdated, and it became hard trying to figure out what was old information from new. My goal will be to have step by step instructions, as well as explanations as to what you are doing, for building a Flask web server on Google App Engine. I will also throw in some explanations on how to get your own personal URL for the site (I used NameCheap), and how to link the code you deploy to App Engine to GitHub so you can push to both at the same time. This will all be under the pre-tense of building your own personal website. Stay tuned...

So why did I build this site? The more and more I have gotten into programming, the quicker I have realized that I learn by doing. There is only so much I can get out of reading a tutorial or typing some lines on Codecademy (not that I dislike Codecademy, I love the site and visit it often). At some point you just have to really dive in. This site was my version of diving into web development. I had learned about building a basic web server and the power of Flask before, but I had absolutely no knowledge of building websites and figured this would be the perfect opportunity to learn. Throw in the fact that I learned a bit about Google App Engine, and I would say that it was an overall great experience that was well worth while.

The other reason I built this site is because every legitimate, boss programmer has his or her own website. And I was ready to become a boss programmer.

If you are working on building your own website and have any questions, feel free to shoot me an e-mail. I can't guarantee that I will have an answer, but I would love to try and help.
